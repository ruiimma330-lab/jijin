# Universal Agent Problem Solving

## Overview

这是一个**通用思维模式 Skill**，目标不是记录某个具体系统怎么操作，而是把 Agent 做复杂任务时最稳定的一套方法抽象出来：

1. 把模糊需求变成可验证目标。
2. 先查真实状态和根因，再动手修复。
3. 小步改动，一次只验证一个假设。
4. 每一步都留下证据：日志、命令、输出、文件路径、测试结果。
5. 修复后做端到端验证，而不是只看“命令没报错”。
6. 把重复错误升级为检查项、脚本、监控或新 Skill。

核心原则：**不要猜，不要跳步，不要用“应该/大概”代替验证。**

---

## When to Use

使用这个 Skill 的场景：

- 用户给出复杂目标，需要拆解执行。
- 遇到报错、异常、失败、超时、卡死、不稳定、结果不符合预期。
- 需要修改代码、脚本、配置、自动化流程或运维任务。
- 任务涉及多个环节：前端、后端、数据库、浏览器、API、定时任务、第三方服务、文件系统等。
- 之前尝试过但失败，或者“今天又失败了”。
- 需要把一次性处理沉淀成可复用 SOP / Skill / watchdog。

不适用或慎用：

- 简单问答、无需验证的写作润色。
- 用户明确只要脑暴，不要求执行。
- 无法获取任何证据且用户也无法提供上下文；此时应明确假设和不确定性。

---

## Mental Model: Evidence → Hypothesis → Minimal Change → Verification

任何问题都按这个闭环处理：

```text
收集证据 → 定位边界 → 形成假设 → 最小改动 → 验证结果 → 沉淀防复发
```

### 不允许的做法

- 看见一个症状就直接修。
- 同时改多个变量，然后不知道哪个生效。
- 用经验判断“应该是 X”，但没有日志/接口/代码/页面证据。
- 只验证局部，不验证用户真正关心的最终结果。
- 修完不留记录，导致下次重复排障。

### 允许的快速处理

生产事故或用户急需恢复时，可以先做**可逆的止血动作**，但必须满足：

1. 动作低风险、可回滚。
2. 明确说明这是止血，不是根因修复。
3. 止血后继续查根因。
4. 最终补上长期防复发措施。

## Phase 0: Frame the Task

先把用户的话转成明确任务。

### 0.1 识别真正目标

问自己：

- 用户要的是解释、修复、交付文件、跑通流程，还是长期稳定？
- 成功标准是什么？
- 哪些动作有副作用？是否需要用户明确确认？
- 哪些上下文可以用工具查到，不该问用户？

### 0.2 产出执行清单

复杂任务创建 todo，且只允许一个 `in_progress`。

模板：

```text
1. 收集状态/日志/输入输出
2. 定位失败边界和根因
3. 制定最小修复
4. 执行修复
5. 验证最终结果
6. 总结和沉淀防复发
```

### 0.3 明确约束

记录：

- 不能泄露的凭据、token、密码。
- 不能动的端口、文件、进程、数据表。
- 用户偏好的输出格式。
- 是否要求实发通知、真实提交、线上变更。

---

## Phase 1: Gather Ground Truth

**先拿真实证据。**

### 1.1 查运行状态

常见检查项：

- 当前时间、时区、机器、用户、工作目录。
- 进程、端口、服务状态、定时任务状态。
- 最新日志、错误堆栈、退出码、请求 ID。
- 配置文件实际内容，而不是记忆里的配置。
- 外部接口真实返回，而不是文档猜测。
- UI/前端问题要查页面、DOM、接口、前端代码或截图。

### 1.2 查输入和输出

每个任务都要确认：

- 输入数据从哪里来？格式是什么？是否过期？
- 中间产物在哪里？是否存在？更新时间？
- 输出写到哪里？用户如何看到？
- 成功/失败的判定字段是什么？

### 1.3 多组件系统按边界排查

对每个边界都要收证据：

```text
调用方 → 脚本/服务 → 浏览器/API/数据库 → 目标系统 → 通知/展示
```

每一层问：

- 请求是否发出？
- 参数是否正确？
- 返回是否成功？
- 错误是在这一层产生，还是下一层传回？
- 是否存在重试、超时、缓存、异步延迟？

## Phase 2: Reproduce and Isolate

### 2.1 尽量复现

优先用最小命令复现：

```bash
# 示例：只跑失败的单元/脚本/接口，不要一上来跑全量
python script.py --dry-run
pytest tests/test_x.py::test_case -v
curl -i https://example.com/health
```

复现时记录：

- 命令。
- 时间。
- 输入参数。
- 输出/日志。
- 退出码。

### 2.2 隔离变量

一次只变一个变量：

- 换输入，不换代码。
- 换环境，不换输入。
- 换账号，不换脚本。
- 换接口路径，不换请求体。

如果同时改三件事，即使成功也不知道原因。

### 2.3 找“工作样本”对比

找同项目里能正常工作的相似流程，对比：

- 参数差异。
- 权限差异。
- 环境变量差异。
- 超时和重试差异。
- 数据结构差异。
- 依赖版本差异。

---

## Phase 3: Root Cause Before Fix

### 3.1 根因陈述格式

修复前必须能写出一句话：

```text
我认为根因是 <具体组件/代码路径/配置> 在 <具体条件> 下 <具体失败行为>，证据是 <日志/返回/代码/复现结果>。
```

例子（通用化）：

```text
我认为根因是任务执行脚本只检查了 HTTP 健康接口，但真实业务使用 WebSocket 会话；健康接口返回 200 时，会话层仍可能不可用。证据是任务日志显示连接/超时失败，而健康检查日志仍显示通过。
```

### 3.2 区分症状、直接原因、根因

| 层级 | 示例 | 是否足够修复 |
|---|---|---|
| 症状 | 用户没收到通知 | 不够 |
| 直接原因 | 通知脚本退出码 1 | 不够 |
| 根因 | socket stale 且启动脚本没有重试/自愈 | 可修 |

### 3.3 三次失败规则

如果连续三次修复都没解决：

1. 停止继续堆补丁。
2. 回到证据收集。
3. 重新画系统边界。
4. 考虑是不是架构模式错了。
5. 必要时向用户汇报“继续小修不划算，需要重构某层”。

---

## Phase 4: Plan the Minimal Fix

### 4.1 修复策略优先级

按稳定性排序：

1. **消除依赖**：能不用不稳定环节就不用。
2. **前置检查**：失败前发现条件不满足。
3. **自动自愈**：可安全恢复的状态自动恢复。
4. **重试退避**：只对瞬时错误重试，且有上限。
5. **失败通知**：失败也必须可见，不能静默。
6. **人工介入**：高风险或无法自动判断时升级给人。

### 4.2 最小改动原则

每次改动只回答一个问题：

- 这个改动验证哪个假设？
- 如果成功，能证明什么？
- 如果失败，下一步查什么？

避免“顺手优化”。顺手优化会污染因果链。

### 4.3 可回滚设计

执行前准备：

- 备份原文件。
- 记录原配置。
- 使用新文件旁路，而不是覆盖核心流程。
- 定时任务先加低频或手动验证。
- 数据写入先 dry-run 或写测试位置。

---

## Phase 5: Implement With Tests or Checks

### 5.1 代码变更：优先 TDD

Bug 修复：

1. 写一个能复现 bug 的失败测试。
2. 看它失败，确认不是测试写错。
3. 写最小代码让它通过。
4. 跑相关测试和全量回归。

如果是脚本/运维任务，没有单元测试，也要做等价检查：

- `bash -n script.sh`
- `python -m py_compile file.py`
- `--dry-run`
- 对假输入跑一次。
- 对最新真实日志跑解析器。

### 5.2 配置变更：验证加载路径

配置类问题必须验证：

- 改的是实际生效文件，不是副本。
- 服务/进程确实重载了新配置。
- live PID / 启动命令 / 工作目录符合预期。
- 真实请求命中新配置。

### 5.3 自动化变更：拆执行和通知

通用可靠模式：

```text
执行脚本 → 写结构化 result.json/log → 通知脚本读取 result → 发送固定模板
```

原则：

- 执行失败也必须写 result。
- 通知不依赖模型推理。
- 通知内容由脚本固定模板生成。
- 发送通道要有重试和自愈。
- 过期结果要告警，不能当成功。

---

## Phase 6: Verify End-to-End

### 6.1 不只看退出码

成功至少要验证两层：

1. **局部成功**：命令退出码、日志、result 文件。
2. **用户可见成功**：页面/表格/API/消息/文件确实出现了正确结果。

### 6.2 验证清单

- [ ] 错误已复现或找到历史证据。
- [ ] 根因有日志/代码/接口证据支持。
- [ ] 修复只改必要范围。
- [ ] 语法检查通过。
- [ ] 单元测试/脚本 dry-run/真实小流量验证通过。
- [ ] 端到端结果正确。
- [ ] 失败路径也验证过：失败会通知或返回明确错误。
- [ ] 没有泄露凭据。
- [ ] 用户关心的最终状态已确认。

### 6.3 失败路径验证

可靠系统必须验证失败路径：

- 输入缺失时会怎样？
- 上游超时时会怎样？
- 通知通道不可用时会怎样？
- 结果文件过期时会怎样？
- 重试次数耗尽时会怎样？

如果失败路径不通知，自动化就会“假装没事”。

---

## Phase 7: Report Clearly

给用户汇报时按这个格式：

```text
结论：已恢复/已修复/部分完成/未完成。

原因：一句话说根因。
证据：列 2-4 条关键日志/状态/路径/时间。
处理：做了哪些动作。
验证：怎么证明已经好。
防复发：加了哪些检查/重试/通知/文档。
剩余风险：还有什么不能完全自动化。
```

避免：

- 大段过程流水账。
- 用“应该好了”。
- 只说做了什么，不说验证结果。
- 隐藏失败和不确定性。

---

## Phase 8: Prevent Recurrence

每次问题修复后，问：

- 这个错误能不能提前检测？
- 下次失败能不能自动恢复？
- 不能恢复时能不能立即通知？
- 这个经验是否应该写入 Skill？
- 是否需要把手工步骤变成脚本？
- 是否需要把日志改成结构化 JSON？

### 8.1 Watchdog 模式

适合周期任务和自动化：

```text
定时执行 → 检查最新结果 → 判断过期/失败 → 自愈/补跑 → 通知
```

关键点：

- watchdog 本身尽量不依赖复杂模型。
- 空输出表示无事发生；非空输出表示需要通知。
- 非零退出要报警。
- 每天/每轮要有防重复标记，避免刷屏。

### 8.2 结构化结果文件

建议每个自动化脚本输出：

```json
{
  "ok": true,
  "started_at": "...",
  "finished_at": "...",
  "input": {},
  "output": {},
  "error": "",
  "log": "/path/to/log"
}
```

这样通知、检查、补跑都可以不读自然语言日志。

### 8.3 Skill 沉淀标准

满足任一条件就考虑写 Skill：

- 同类问题可能再次出现。
- 排查超过 5 个工具调用。
- 中间踩了坑。
- 需要特殊命令顺序。
- 用户明确要求“记住这个方法”。

Skill 应写方法论和可复用流程，不要写一次性任务进度。

## Error Handling Playbook

### A. 超时 / 卡住

1. 查超时发生在哪一层：连接、请求、读取、解析、写入、通知。
2. 找是否有部分成功输出。
3. 检查是否缺少 first-byte timeout、idle timeout、overall timeout 的区分。
4. 给外部调用加明确超时和错误信息。
5. 对瞬时错误重试；对状态损坏先自愈再重试。
6. 验证重试不会重复写入或造成副作用。

### B. 状态看似正常但业务失败

1. 不要只信 health check。
2. 对比 health check 覆盖的层和真实业务使用的层。
3. 增加更接近真实业务的轻量检查。
4. 避免检查本身过重，导致把系统检查坏。
5. 记录“假健康”条件，下一次前置处理。

### C. 结果过期 / 定时任务没跑

1. 查调度器是否触发。
2. 查任务退出码。
3. 查 stdout/stderr 重定向日志。
4. 查结果文件更新时间。
5. 如果执行和通知拆开，分别查两段。
6. 加过期判断：超过阈值必须告警。

### D. 通知失败

1. 通知脚本不能依赖模型生成内容。
2. 发送通道要能自愈：stale socket、进程未启动、网络短故障。
3. 消息编码要稳定，尤其中文。
4. 失败时写本地日志，避免完全丢失。
5. 通知内容要包含任务、时间、状态、错误摘要、日志路径。

### E. 配置改了但没生效

1. 查实际加载的配置路径。
2. 查进程启动命令和工作目录。
3. 查环境变量覆盖。
4. 重启/热加载后验证 live PID 或启动时间变化。
5. 用真实请求验证，不只看文件内容。

### F. 接口返回和前端显示不一致

1. 抓真实接口返回。
2. 查前端使用哪个接口/字段。
3. 查缓存、本地存储、CDN、构建产物。
4. 查是否有多个配置源。
5. 用页面或截图验证最终显示。

### G. 数据写入风险

1. 先 dry-run 生成写入计划。
2. 明确写哪些字段/单元/表/记录。
3. 写前备份或读取原值。
4. 写后读取目标位置验证。
5. 避免重复写造成累加或覆盖错误。

## Quality Gates

### Before Acting

- [ ] 我知道用户真正要的结果。
- [ ] 我知道哪些动作有副作用。
- [ ] 我知道从哪里拿真实证据。

### Before Fixing

- [ ] 我读过完整错误信息。
- [ ] 我知道失败发生在哪一层。
- [ ] 我有根因假设和证据。
- [ ] 我没有同时改多个变量。

### Before Reporting Done

- [ ] 我验证了用户可见结果。
- [ ] 我验证了失败路径或至少说明未验证。
- [ ] 我没有泄露凭据。
- [ ] 我说明了防复发措施和剩余风险。

## Common Pitfalls

1. **把记忆当事实。** 记忆只提供线索；当前状态必须用工具查。
2. **只查健康接口。** 健康接口可能没覆盖真实业务路径。
3. **没有失败通知。** 自动化最危险的失败是静默失败。
4. **修复后只看退出码。** 必须验证最终输出。
5. **一次改太多。** 成功也不知道原因，失败也不知道哪错。
6. **不区分临时止血和长期修复。** 止血后必须继续补防复发。
7. **把业务细节写进通用 Skill。** 通用 Skill 写思维和流程；具体域名、账号、表名、端口放项目 Skill 或安全存储。
8. **没有防重复。** 自动补跑、自动通知、自动重启都要有节流和幂等设计。
9. **忽视编码和环境。** 定时任务、系统服务、交互 shell 的 PATH/编码/工作目录经常不同。
10. **让 Agent 自己审自己。** 重要代码/方案应使用独立上下文或至少重新读 diff 自审。

---

## Final Rule

```text
真实证据 > 经验判断
端到端验证 > 局部成功
根因修复 > 症状补丁
可复用沉淀 > 一次性救火
```
